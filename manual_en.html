0. Dockerized Core Architecture

The entire M-IT DCIM system is designed to run in an isolated container environment. This guarantees audit compliance, easy scalability, and consistent runtime environments.

Container Communication (Internal Bridge)

The main application (dcim_manager) communicates with database and AI sidecars over the internal Docker network. Direct IP addresses within the stack are obsolete; addressing is handled via service names.

  • Database Service: Reachable internally at db:5432.
  • AI Engine: Interactive NLP requests flow to ai_engine:11434.
  • Web Interface: Port mapping routes host 5000 to container 5000.

Volume Persistence & Data Security

All volatile container data is secured via host mounts. During a container update or restart, your DC plans and databases remain untouched.

Critical Paths:

  • ./db_data → Persistent PostgreSQL database raw data.
  • ./backups → Automated SQL dumps and archive snapshots.
  • ./ai_models → Storage for local LLM weights (llama3, etc.).

Hardware Requirements for Smart Features

To leverage the full potential of interactive real-time visualizations (such as live power consumption, thermal state, humidity, A/B redundancy checks, and PUE calculation), network-enabled Smart PDUs and environmental sensors with an active SNMP interface are mandatory! Without this physical hardware foundation, the system will only display your manually maintained, static database values.
